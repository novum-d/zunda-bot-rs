# 誕生日未登録ユーザー向けリマインド機能 要件整理

## このドキュメントの位置づけ

この Issue は実装 Issue ではなく、誕生日未登録ユーザー向けリマインド機能を安全に実装できる単位へ分割するための要件整理 Issue である。

現時点では、Bot の挙動変更、データモデル変更、定期実行追加、インフラ変更、本番環境変数追加は行わない。

## 背景

誕生日未登録ユーザーへのリマインド機能は、以下の要素を同時に含む可能性がある。

- 未登録ユーザーを判定するロジック
- リマインド通知の投稿先と投稿タイミング
- 通知停止 / 再開のユーザー操作
- 通知状態を保存するデータモデル
- 定期実行の起動方式
- Cloud Scheduler / Cloud Run などの運用設計

これらを 1 Issue でまとめて自動実装すると、Codex 運用ルール上のスコープを超えるため、後続 Issue に分割して扱う。

## この Issue でやること

- リマインド機能の実装前提を整理する
- 通知条件、抑止条件、停止 / 再開操作、運用方式を仕様として分離する
- Codex が自動実装できない項目を明確にする
- 後続の小さな実装 Issue に分割できる状態にする

## この Issue でやらないこと

- リマインド機能の一括実装
- `src/**` の挙動変更
- `tests/**` の追加または変更
- DB schema / データモデル変更
- 新しい定期実行 worker の追加
- Bot チャンネル投稿処理の実装
- 通知停止 / 再開 UI の実装
- Cloud Scheduler / Cloud Run / デプロイ構成の変更
- 本番設定や環境変数の追加

## 仕様案

### 目的

誕生日が未登録のアクティブユーザーに対して、Bot 専用チャンネルでメンション付きのリマインドを行い、`/birth signup` への導線を提供する。

通知は便利さよりもスパム抑止を優先し、以下を前提にする。

- 初回リマインドはアクティブ化から 1 日遅延する
- リマインド間隔は段階的に伸ばす
- 5 回送信したら自動停止する
- ユーザー単位で通知停止 / 再開できる
- 再起動やスキャン再実行で重複送信しない

### 対象ユーザー

アクティブユーザーは、直近 7 日以内に以下のいずれかを行ったユーザーとする。

- メッセージ送信
- スラッシュコマンド実行

ただし、後続 Issue では以下を明確にする必要がある。

- Bot ユーザーを対象外にする条件
- 退会済みユーザーを対象外にする条件
- 「Bot が DB 上で認識済みの未登録メンバー」だけを対象にするか、Discord API から取得した未登録メンバーも対象にするか

### データモデル案

`guild_member` に以下の状態を保存する案とする。

- `last_active_at`: 最終アクティブ日時
- `last_reminded_at`: 最終リマインド送信日時
- `next_remind_at`: 次回リマインド予定日時
- `remind_count`: リマインド送信回数
- `is_remind_opt_out`: ユーザーによる通知停止状態

schema 変更は人手承認が必要なため、この Issue では migration を追加しない。

### リマインド間隔

初回アクティブ時は `next_remind_at = last_active_at + 1日` とする。

送信後は前回送信時刻を基準に、以下の間隔で `next_remind_at` を更新する。

| 送信回数 | 次回までの間隔 |
| --- | --- |
| 1 回目 | 1 日 |
| 2 回目 | 3 日 |
| 3 回目 | 7 日 |
| 4 回目 | 14 日 |
| 5 回目 | 30 日 |

`remind_count >= 5` の場合は送信しない。

### 送信条件

以下をすべて満たす場合のみ送信する。

- 誕生日未登録
- `is_remind_opt_out = false`
- `remind_count < 5`
- `last_active_at` が 7 日以内
- 現在時刻が `next_remind_at` 以降
- `last_reminded_at` が存在する場合、前回送信から 24 時間以上経過している

### 送信後更新

送信成功後に以下を更新する。

- `last_reminded_at`
- `remind_count`
- `next_remind_at`

冪等性と再起動時のスパム抑止のため、送信条件と送信後更新は同じ usecase 内で扱う。

### 通知内容

投稿先は Bot 専用チャンネルとし、チャンネル ID は `BOT_CHANNEL_ID` で指定する案とする。

1 ユーザーにつき 1 投稿とし、複数ユーザーへ送信する場合は投稿間に 1 から 3 秒のディレイを入れる。

本文案:

```text
まだ誕生日が登録されていないのだ！
よければ `/birth signup` から登録してほしいのだ！

通知が不要な場合は下のボタンから止められるのだ
```

文体は「のだ」で統一する。

### 停止 / 再開

停止は通知メッセージ上のボタンで行う案とする。

- 本人が押下した場合、`is_remind_opt_out = true` に更新する
- 本人にはエフェメラルで「通知を停止したのだ！」と返信する
- 停止後はボタンを無効化する
- 他人が押下した場合、エフェメラルで「この操作は自分のリマインドにのみ使えるのだ」と返信する

再開は `/birth remind resume` で行う案とする。

- `is_remind_opt_out = false` に更新する
- `next_remind_at = now + 1日` に更新する
- エフェメラルで「リマインドを再開したのだ！」と返信する
- 再開直後には即時送信しない

### 起動契機

イベント処理を主、スキャン処理を補助とする。

イベント処理:

- メッセージ送信時に `last_active_at` を更新する
- スラッシュコマンド実行時に `last_active_at` を更新する
- `now < next_remind_at` の場合は早期 return する
- 条件を満たす場合のみリマインド送信する

スキャン処理:

- 7 日に 1 回、直近 7 日以内にアクティブなユーザーを確認する
- `next_remind_at` と送信後更新により重複送信を防ぐ
- アプリ内部 cron は追加せず、Cloud Scheduler からの起動を前提にする

### Cloud Scheduler / Cloud Run 案

運用方式は以下を候補にする。

- Cloud Scheduler から Cloud Run へ HTTP POST する
- 専用エンドポイントは `/internal/reminder/scan` とする
- 内部利用のみとし、認証ヘッダまたは IAM で保護する

インフラ、デプロイ、本番設定は現行ルール上この Issue では変更しない。

## 自動実装できない項目

以下は現行ルール上、別 Issue または人手承認が必要である。

- データモデル / schema 変更
- 新しい定期実行の追加
- インフラ / デプロイ / Cloud Scheduler 関連変更
- 本番設定や環境変数の追加
- 新しい依存関係の追加
- `src/main.rs` など、現行 AGENTS.md の許可パス外で必要になる配線変更

## 分割方針

後続 Issue は、1 Issue につき 1 つの責務に絞る。

1. 要件整理と仕様確定
   - 本ドキュメントで扱う。
   - 実装は行わない。
2. 通知対象判定ロジックの実装
   - 誕生日未登録、アクティブ条件、5 回停止、24 時間抑止を純粋関数として実装する。
   - `src/usecase/**` または `src/models/**` に小さく実装し、`tests/**` で検証する。
   - schema 変更を含めない。
3. schema 変更の承認付き Issue
   - migration、モデル、リポジトリ更新を含める。
   - `last_active_at`、`last_reminded_at`、`next_remind_at`、`remind_count`、`is_remind_opt_out` を追加する。
4. アクティブ更新処理の実装
   - メッセージ送信とスラッシュコマンド実行で `last_active_at` を更新する。
   - 必要な `src/main.rs` 配線変更を許可範囲として明記する。
5. 通知投稿処理の実装
   - `BOT_CHANNEL_ID` を読み、Bot 専用チャンネルへ通常メッセージを送信する。
   - ボタン付きメッセージと投稿間ディレイを実装する。
6. 通知停止ボタンの実装
   - 本人のみ操作可能にする。
   - 停止時のエフェメラル返信とボタン無効化を実装する。
7. 再開コマンドの実装
   - `/birth remind resume` または既存コマンド構成に合わせた再開操作を実装する。
   - 再開直後に即時送信しないことをテストする。
8. 定期スキャン起動方式の承認付き Issue
   - `/internal/reminder/scan` の追加可否、認証方式、Cloud Scheduler 設定を決める。
   - インフラ変更が必要な場合は人手で扱う。

## 後続 Issue 作成時のチェック項目

実装 Issue を作る場合は、以下を本文に明記する。

- 変更対象の責務
- 変更を許可するファイルまたはディレクトリ
- schema 変更の有無
- 環境変数追加の有無
- インフラ / デプロイ変更の有無
- 期待する通知対象条件
- 期待する通知抑止条件
- 必要なテスト観点
- `src/main.rs` や `db/migrations/**` など、通常の許可パス外を触る必要があるか

実装系 Issue では、原則として `src/**` の挙動変更と `tests/**` の検証を含める。
