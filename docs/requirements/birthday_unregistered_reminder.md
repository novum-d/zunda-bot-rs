# 誕生日未登録ユーザー向けリマインド機能 要件整理

## このドキュメントの位置づけ

この Issue は実装 Issue ではなく、誕生日未登録ユーザー向けリマインド機能を安全に実装できる単位へ分割するための要件整理 Issue である。

現時点では、Bot の挙動変更、データモデル変更、定期実行追加、インフラ変更、本番環境変数追加は行わない。

## 背景

誕生日未登録ユーザーへのリマインド機能は、以下の要素を同時に含む可能性がある。

- 未登録ユーザーを判定するロジック
- リマインド通知の投稿先と投稿タイミング
- 通知停止 / 再開のユーザー操作
- 通知状態を保存するデータモデル
- 定期実行の起動方式
- Cloud Scheduler / Cloud Run などの運用設計

これらを 1 Issue でまとめて自動実装すると、Codex 運用ルール上のスコープを超えるため、後続 Issue に分割して扱う。

## この Issue でやること

- リマインド機能の実装前提を整理する
- 通知対象判定、通知頻度、停止条件、再開条件を後続 Issue の入力として使える粒度で文章化する
- Codex が自動実装できない項目を明確にする
- 後続の小さな実装 Issue に分割できる状態にする

## この Issue でやらないこと

- リマインド機能の一括実装
- `src/**` の挙動変更
- `tests/**` の追加または変更
- DB schema / データモデル変更
- 新しい定期実行 worker の追加
- Bot チャンネル投稿処理の実装
- 通知停止 / 再開 UI の実装
- Cloud Scheduler / Cloud Run / デプロイ構成の変更
- 本番設定や環境変数の追加

## 確定仕様

後続 Issue では、ここで確定した仕様を実装前提として扱う。

### 通知対象判定

リマインド対象は、Bot が認識済みのギルドメンバーのうち、`guild_member.birth` が `NULL` のメンバーとする。

対象に含める条件は以下の通り。

- `guild_member` に存在する
- 対象ギルドが現在も Bot から取得できる
- 対象メンバーが現在も対象ギルドから取得できる
- `birth` が `NULL`
- 通知停止状態ではない

対象外にする条件は以下の通り。

- `birth` が登録済み
- Bot が所属しなくなったギルド
- ギルドから退会済み、または Discord API から取得できないメンバー
- Bot ユーザー
- 通知停止状態のユーザー

未登録ユーザーを「Discord サーバー上の全メンバー」ではなく「Bot が認識済みのメンバー」とする理由は、既存のギルド同期処理と `guild_member` を基準にすると、通知対象判定をアプリケーション内で再現しやすく、後続実装のスコープを小さくできるためである。

### 通知頻度

リマインドはギルド単位で月 1 回までとする。

同一ユーザーへの再通知は、前回リマインドから 30 日以上経過している場合のみ許可する。誕生日を登録したユーザーは `birth` が `NULL` ではなくなるため、次回以降のリマインド対象から外れる。

実行時刻は既存の誕生日通知 worker と同じく `Asia/Tokyo` を基準にする。具体的な実行方式は後続 Issue で決めるが、同日に複数回起動しても同じユーザーへ重複通知しないことを実装条件に含める。

### 通知停止条件

ユーザー単位でリマインド停止状態を持つ。停止状態のユーザーには、誕生日が未登録のままでもリマインドを送らない。

停止条件は以下の通り。

- ユーザーが明示的にリマインド停止操作を行った
- ユーザーが誕生日を登録した
- ユーザーが対象ギルドから退会した

誕生日登録による停止は、既存の `birth` 値で判定できる。明示的な停止状態と最終リマインド日時を永続化する場合は、現在の schema だけでは表現できないため、別 Issue でデータモデル変更の承認を取る。

### 再開条件

明示的にリマインド停止したユーザーは、ユーザー自身の再開操作があるまでリマインド対象へ戻さない。

再開条件は以下の通り。

- ユーザーが明示的にリマインド再開操作を行った
- 停止状態が解除されている
- `birth` が `NULL`
- 対象ギルドと対象メンバーが現在も Bot から取得できる

誕生日登録済みユーザーが `/birth reset` などで誕生日を解除した場合、`birth` は `NULL` になる。ただし、明示的なリマインド停止状態が残っている場合は再開操作があるまで対象外とする。

### 通知内容と投稿先

通知は、未登録ユーザー本人に誕生日登録を促す内容とする。メッセージには `/birth signup` で登録できることを含める。

投稿先の優先順位は後続 Issue で実装するが、仕様上は以下の順序とする。

1. 将来の設定値で指定されたチャンネル
2. 既存の誕生日通知と同じく `一般` または `general` チャンネル
3. 投稿可能なチャンネルが見つからない場合は通知をスキップし、エラーログまたは警告ログに残す

チャンネル設定を永続化する場合は schema または設定管理の変更が必要になるため、承認付き Issue に分離する。

## 実装依頼に必要な前提

後続の実装 Issue には、少なくとも以下を明記する。

- 対象ギルドを単一ギルドに限定するか、Bot が参加する全ギルドにするか
- 明示的な停止 / 再開 UI を `/birth` のサブコマンド、ボタン、または別コマンドのどれで提供するか
- 最終リマインド日時と停止状態をどのデータ構造に保存するか
- 月 1 回判定を worker 内で行うか、外部スケジューラ側で制御するか
- 投稿先チャンネル設定を初期実装に含めるか
- Bot ユーザー判定を Discord API 取得時に行うか、永続化済みデータに属性を追加して行うか

## 自動実装できない項目

以下は現行ルール上、別 Issue または人手承認が必要である。

- データモデル / schema 変更
- 新しい定期実行の追加
- インフラ / デプロイ / Cloud Scheduler 関連変更
- 本番設定や環境変数の追加
- 新しい依存関係の追加

## 分割方針

後続 Issue は、1 Issue につき 1 つの責務に絞る。

1. 要件整理と仕様確定
   - 本ドキュメントで扱う。
   - 実装は行わない。
2. 通知対象判定ロジックの実装
   - `guild_member.birth IS NULL` を中心に、現在も取得可能なギルド / メンバーだけを対象にする。
   - Bot ユーザーは対象外にする。
   - 可能であれば `src/usecase/**` または `src/data/**` に小さく実装し、`tests/**` で検証する。
3. 通知停止 / 再開の操作設計
   - 明示的な停止 / 再開操作の UI を決める。
   - 停止状態の永続化が必要なため、schema 変更 Issue に分離する。
4. 定期実行方式の決定と承認
   - 既存 worker へ組み込むか、外部スケジューラから起動するかを決める。
   - Cloud Scheduler / Cloud Run などのインフラ変更が必要な場合は、人手承認付き Issue とする。
5. schema 変更の承認付き Issue 作成
   - 通知停止状態や最終リマインド日時の保存が必要なため作成する。
   - migration、モデル、リポジトリ、テストを含める。

## 後続 Issue 作成時のチェック項目

実装 Issue を作る場合は、以下を本文に明記する。

- 変更対象の責務
- 変更を許可するファイルまたはディレクトリ
- schema 変更の有無
- 環境変数追加の有無
- インフラ / デプロイ変更の有無
- 期待する通知対象条件
- 期待する通知抑止条件
- 必要なテスト観点

実装系 Issue では、原則として `src/**` の挙動変更と `tests/**` の検証を含める。

## 未確定事項

後続 Issue の前に、以下を決める必要がある。

- 停止 / 再開 UI の具体的な操作方法
- 停止状態と最終リマインド日時の保存先
- 投稿先チャンネル設定を持つか、既存の `一般` / `general` 探索だけで開始するか
- 既存の誕生日通知 worker に含めるか、別の定期実行にするか
- 月 1 回の実行日を固定するか、前回実行からの経過日数で判定するか
- 複数ギルドに同じユーザーがいる場合、停止状態をギルド単位にするかユーザー全体にするか

## 受け入れ条件

- この Issue が要件整理 / 分割 Issue であり、実装 Issue ではないことが明確である
- 通知対象判定、通知頻度、停止条件、再開条件が文章化されている
- 実装依頼に必要な前提が文章化されている
- 未確定事項が明示されている

- 後続の小さな実装 Issue に分割できる観点が整理されている
